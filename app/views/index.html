Documentation

Start a new sprint

To create a new sprint:

  • Duplicate the current sprint app/views/SPRINTNAME folder and make sure it is named for the new sprint (sprintn+1 )
  • Duplicate the current sprint app/assets/config/SPRINTNAME folder and make sure it is named for the new sprint (sprintn+1 )
  • In the new sprint's app/assets/config/SPRINTNAME/changelog.md file remove the old data and add an entry for the creation of the new sprint
  • Update the sprintName at the top of the new sprint's app/assets/config/SPRINTNAME/routes.js file

Then make this the new current sprint:

  • Change the currentSprint variable in the root app/config.js file
  • Update this page's (app/views/index.html) content to reflect the new current sprint and move the old one to the “previous” section

Current sprint

The current sprint prototype can always be reached via use of the word current in place of the sprint name in the url

Instead of using this:
http://localhost:3000/sprint17/section-name/page-name

You can reference this:
http://localhost:3000/current/section-name/page-name

This means a consistent url can be added to user stories without worrying that you are pointing to an old sprint.

Accessing an older sprint than the one set as current in the app/config.js will show a warning banner at the top of the page to alert you to the fact you might be viewing an outdated screen.

Changelogs

Each sprint has an associated changelog, a markdown file which lists the UR-related changes made to the prototype during that period. This is pulled into the list above.

This means:

  • the team and stakeholders have an easy to access overview of what has changed
  • the UR can see if anything has happened which will affect how they conduct a session
  • changelogs can be collated and easily ported into an assessment document to show how a service was improved over time

(more information about changelogs can be found in the readme: app/assets/config/documentation/changelog.md)
